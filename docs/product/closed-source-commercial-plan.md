# 闭源商用计划书

版本：v0.1  
日期：2026-06-26  
项目代号：微信多开商用工作台  
商业形态：闭源主程序 + 独立开源助手组件 + 云端授权服务

## 1. 执行摘要

本项目计划推出一款 Windows 桌面端微信多开商用工作台，面向私域运营、客服销售、中小商家和门店导购。产品用闭源主程序提供商业化体验，包括多账号窗口管理、快捷话术、截图工具、授权会员、团队配置和更新服务；底层涉及 GPLv3 项目能力的部分以独立开源助手组件形式隔离分发，确保闭源商业化边界清晰。

商业核心不是售卖“补丁”，而是售卖一个高频工作台：节省客服回复时间、降低多账号切换成本、统一团队话术和提供持续兼容更新。

## 2. 商业定位

一句话定位：

面向中小商家和私域团队的 Windows 微信多开与快捷话术工作台。

核心卖点：

- 多微信账号集中管理。
- 高频话术一键复制/粘贴。
- 截图与窗口操作集中在一个壳内。
- 授权、更新、客服支持稳定可持续。
- 合规隔离开源组件，闭源主程序可商业运营。

差异化：

- 不只做多开工具，而是做客服/销售工作流。
- 不以灰色自动化为卖点，降低封号、投诉和支付渠道风险。
- 有明确开源许可证声明，减少 GPL 争议。

## 3. 产品架构与闭源边界

### 3.1 架构分层

闭源主程序：

- UI 工作台。
- 账号和窗口状态管理。
- 快捷话术数据库。
- 截图工具。
- 授权会员。
- 自动更新。
- 日志诊断。
- 云端 API 对接。

独立开源助手组件：

- 多开相关底层能力。
- 可选防撤回补丁能力。
- 命令行或本地 IPC 接口。
- 独立仓库维护。
- 保留 GPLv3 许可证。
- 提供源码下载地址和构建说明。

云端服务：

- 用户账号。
- 激活码。
- 设备绑定。
- 版本更新。
- 公告与配置。
- 团队席位管理。

### 3.2 GPLv3 合规策略

必须遵守：

- 不把 GPL 源码复制进闭源主程序。
- 不把 GPL 项目编译成闭源主程序的一部分。
- 不通过静态链接或动态链接把 GPL 代码作为闭源程序内部库。
- GPL 助手组件独立分发、独立运行、独立许可证声明。
- 对用户提供 GPL 助手组件源码获取方式。
- 对修改过的 GPL 助手组件公开对应修改源码。

推荐做法：

- 建立独立仓库 `wechat-workbench-helper-open`。
- 保留来源说明：基于 RevokeMsgPatcher 的 GPLv3 代码或思路时，清晰标注。
- 主程序调用助手组件时使用命令行参数或本地 IPC。
- 安装器中把主程序和助手组件作为两个组件呈现。
- 设置页提供“开源组件与许可证”入口。

### 3.3 商业资产边界

可闭源资产：

- 主程序 UI 和交互。
- 账号工作台逻辑。
- 话术库、团队配置、授权逻辑。
- 云端服务。
- 商业品牌、官网、安装器、更新器。

应开源或隔离资产：

- 直接来自 RevokeMsgPatcher 的源码修改。
- 直接复用的 GPL 补丁逻辑。
- 与 GPL 代码形成同一程序作品的派生实现。

## 4. 目标市场

优先市场：

- 私域电商。
- 本地生活商家。
- 中小型客服团队。
- 社群运营人员。
- 微商/导购/咨询销售。

典型客户画像：

- 个人版：单人运营 2-5 个微信号，需要多开和常用回复。
- 专业版：个人或小团队 5-20 个微信号，需要话术库和截图工具。
- 团队版：3-30 人客服团队，需要统一授权、统一话术、员工设备管理。

不建议进入的市场：

- 自动群发软件市场。
- 批量营销和养号工具市场。
- 需要破解微信协议的自动化控制市场。

## 5. 收费模式

### 5.1 版本设计

免费试用：

- 7 天试用。
- 最多 2 个微信窗口。
- 本地话术最多 50 条。
- 不支持导入导出。
- 显示开源组件声明和升级入口。

个人版：

- 19-29 元/月，或 99-149 元/年。
- 最多 5 个微信窗口。
- 话术无限。
- 支持导入导出。
- 支持截图工具。

专业版：

- 39-59 元/月，或 199-299 元/年。
- 最多 10 个微信窗口。
- 高级话术分类。
- 快捷短语栏。
- 自动更新优先通道。
- 本地数据加密。

团队版：

- 299 元/月起，按席位或设备数计费。
- 统一话术配置。
- 员工设备管理。
- 授权后台。
- 专属更新和客服支持。

### 5.2 付费限制点

适合收费的能力：

- 可管理账号数量。
- 话术条数和分类数量。
- 导入导出。
- 团队统一配置。
- 云端模板库。
- 自动更新和技术支持。

不建议作为主要收费点：

- 单纯多开能力。
- GPL 助手组件本身。
- 防撤回补丁本身。

原因：

GPL 组件本身应保持源码可得，商业价值应落在闭源工作台体验、服务和维护上。

## 6. Go-To-Market 计划

### 6.1 第一阶段：种子验证

周期：2-4 周

目标：

- 完成可运行 MVP。
- 找到 20-50 个真实运营用户试用。
- 验证多开、话术、截图是否构成付费理由。

渠道：

- 私域运营社群。
- 中小商家微信群。
- B 站/抖音短演示。
- 小红书效率工具内容。
- 独立官网下载页。

验证指标：

- 安装成功率。
- 首次启动成功率。
- 多开成功率。
- 话术使用频率。
- 试用后主动咨询付费比例。

### 6.2 第二阶段：付费转化

周期：4-8 周

目标：

- 上线个人版和专业版。
- 接入激活码和自动更新。
- 建立售后反馈闭环。

动作：

- 官网提供下载和购买。
- 加入 7 天试用。
- 设置付费限制点。
- 建立更新日志和问题反馈入口。

指标：

- 试用到付费转化率 3%-8%。
- 首月付费用户 50-200。
- 退款率低于 10%。

### 6.3 第三阶段：团队版

周期：8-16 周

目标：

- 上线团队授权后台。
- 提供统一话术配置。
- 争取小团队年付客户。

动作：

- 建立团队管理后台。
- 支持席位和设备绑定。
- 提供企业版报价。
- 针对客服团队做案例页。

指标：

- 团队版客户 5-20 家。
- 月经常性收入稳定增长。
- 客服支持成本可控。

## 9. 风险与控制

### 9.1 许可证风险

风险：

直接使用 GPLv3 源码开发闭源主程序可能导致主程序需要开源。

控制：

- 闭源主程序和 GPL 助手组件分离。
- 修改 GPL 代码时公开对应源码。
- 安装包和官网提供许可证声明。
- 重要发布前做一次开源合规审查。

### 9.2 平台兼容风险

风险：

微信更新后多开、窗口嵌入、防撤回能力失效。

控制：

- 助手组件独立升级。
- 主程序提供非嵌入聚焦模式。
- 不把“永久可用”作为营销承诺。

### 9.3 法务与平台风险

风险：

产品被理解为破坏平台规则或帮助违规营销。

控制：

- 不提供自动群发、批量加好友、机器人营销。
- 营销文案强调效率工具，不强调规避风控。
- 用户协议禁止滥用。

### 9.4 安全与隐私风险

风险：

用户担心软件读取聊天记录、账号信息或上传隐私。

控制：

- 默认不读取聊天数据库。
- 不上传聊天内容。
- 授权只上传设备授权必要信息。
- 提供隐私说明和本地数据删除能力。

### 9.5 商业转化风险

风险：

用户认为多开免费工具很多，不愿付费。

控制：

- 付费价值集中在话术、团队、稳定更新和售后。
- 提供免费试用降低决策成本。
- 用行业话术模板提高首日价值感。

## 10. 财务假设

基础假设：

- 第一阶段个人/专业版为主要收入。
- 第二阶段团队版提高客单价。
- 早期研发以 1-3 人小团队推进。

示例模型：

- 第 1-2 月：验证期，月收入 0-5000 元。
- 第 3-4 月：个人版转化，月收入 5000-30000 元。
- 第 5-6 月：团队版上线，月收入 30000-100000 元。

主要成本：

- 开发人力。
- 代码签名证书。
- 服务器和对象存储。
- 官网和支付通道。
- 客服与售后。
- 法务和开源合规审查。

## 12. 结论

本项目可以按“闭源主程序 + 独立开源助手组件”的方式推进商业化。闭源主程序承担付费价值和用户体验，开源助手组件承担 GPLv3 相关底层能力并保持源码可得。这样既能利用现有开源项目方向，又能保留商业产品的品牌、体验、授权和云服务资产。
